   - 自動重導至管理員儀表板

3. **12.1.1.3：登出流程**
   - 登入後執行登出
   - 驗證 session cookie 清除
   - 確認重導至登入頁面

4. **12.1.1.4：受保護路由 - 未認證重導**
   - 清除認證狀態

   - 驗證被強制重導至登入頁面

   - 生產環境驗證 Secure 標記

7. **12.1.1.7：多次登入 - Session 更新**
   - 登入→登出→再登入
   - 驗證 session token 已更新（不是舊 token）

8. **12.1.1.8：頁面重新整理 - Session 持久化**
   - 登入後重新整理頁面
   - 驗證仍在同一頁面（未被重導至登入）
   - 確認 session 持續有效

   - 建立兩個標籤頁
   - 在第一個登入
   - 驗證第二個無需登入即可訪問受保護路由
   - 確認兩個頁面共享相同 session token

10. **12.1.1.10：無效 Session - 錯誤處理**
    - 手動刪除 session cookie
    - 驗證訪問受保護路由被拒
    - 強制重導至登入頁面

## 決策與設計考量

### 1. Playwright 框架選擇

**決策：** 使用 Playwright 而非 Cypress 或 Selenium

**原因：**
- ✓ 原生 TypeScript 支援
- ✓ 多瀏覽器支援（Chrome、Firefox、Safari）
- ✓ WebSocket 連接監控能力
- ✓ 強大的自動化 API
- ✓ 與 Next.js 良好集成

### 2. Page Object 模式

**決策：** 實現 Page Object 模式而非在測試中直接操作 DOM

**優點：**
- ✓ 提高測試可維護性
- ✓ 減少重複程式碼
- ✓ 便於頁面結構變更時的維護
- ✓ 提高測試可讀性

### 3. Fixtures 策略

**決策：** 建立可重用的認證 fixtures

**優點：**
- ✓ 避免在每個測試重複登入邏輯
- ✓ 確保一致的測試前置條件
- ✓ 自動清理（登出）

### 4. WebSocket 測試方法

**決策：** 多層驗證方式

**實現：**
1. 檢查 WebSocket 連接事件
2. 監控通知 UI 元素
3. 測試 API 端點回應
4. 驗證消息順序和去重

### 5. 並發測試

**決策：** 支援多標籤頁和多用戶並發測試

**原因：**
- ✓ 驗證 session 在多標籤頁間的共享
- ✓ 測試並發連接穩定性
- ✓ 驗證通知多用戶推播
